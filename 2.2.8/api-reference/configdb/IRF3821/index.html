    Packet Filter & NAT¶

    1:1 NAT¶

    1:1 NAT translates entire subnets by replacing the network prefix while preserving the host portion. This is used to resolve IP address conflicts when two networks share the same address range. Variables use the interface prefix pattern: (wan|lan|lan_port[1-7])_.

    The public (virtual) subnet is derived from the interface's _ipaddr and _netmask. Double-sided network mapping (_netmap_ext_*) controls the virtual address range under which the opposite side's devices appear.

    DHCP interfaces

    On interfaces using dhcp or dhcpovpn protocol, the private subnet is derived automatically from the DHCP lease. The _netmap_subnet variable is only used for static interfaces.

    See the 1:1 NAT guide for a complete walkthrough.

    Forwarding¶

    forwarding table¶

    Configuration of IP fowarding

    Defines DNAT (port forwarding) rules. Changes take effect immediately on commit — no trigger variable is needed (unlike packet filter tables which require writing to l3_forward).

    wan_ifname = dev.config_get(["wan_ifname"])["result"][0]["wan_ifname"]
cfg = dev.sess_start()
dev.table_insert("forwarding", cfg, [
    wan_ifname, "tcp", "", "3389", "192.168.10.200",
    "3389", "disabled", "", "RDP access", "enabled", "1", "disabled",
])
dev.sess_commit(cfg)

    See the Port Forwarding guide for SNAT usage and more examples.

    Packet filter¶

    Reload required after table changes

    Modifying the packet filter tables (services, serv_Protocols, selected_services) does not automatically reload the running filter. You must write to l3_forward (for L3 rules) or l2_forward (for L2 rules) to apply changes.

    l3_forward `` 0-64 times: letter, digit . _ - Apply layer 3 packet filter configuration (trigger variable, triggered with Unix timestamp)
